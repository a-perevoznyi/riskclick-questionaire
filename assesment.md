# Technical Assessment

## 1. Project Overview

One of the projects I worked on was a SaaS B2B application for Shopify merchants. The product gave merchants a trade-in feature, so they could offer their customers trade-in options directly in their stores.

One of the business needs was to make subscription pricing more flexible. Different merchants had different needs, so fixed pricing plans were not enough. We needed a solution where pricing rules could be customized without changing the application code every time.

The project was built on the Shopify platform using a **Remix** application. Remix is not the same as Next.js, but it is a modern React full-stack framework with many similar principles around routing, server-client data flow, forms, rendering, and web application structure. In my case, Remix was a good lightweight fit for the product and platform needs.

The frontend used React and JavaScript/TypeScript, and the backend used Node.js. We also used AWS services, including CloudFront for CDN delivery, and CI/CD for deployment automation.

Although this project was built with Remix and not Next.js, I am an experienced full-stack engineer and I work confidently with the main principles behind modern React full-stack applications. Because of that, I am confident I can work with Next.js effectively, even if my direct commercial experience is with Remix rather than Next.js itself.

## 2. My Role & Contribution

## 3. Technical Challenge & Solution

The main technical challenge was how to support flexible pricing for many merchants in a clean and scalable way.

We needed a solution where pricing could be changed through admin settings, without changing and redeploying the main Node.js application each time. At the same time, the solution had to stay simple, fast, and easy to support.

I considered a few approaches.

The first option was to keep pricing logic hardcoded inside the Node.js application. This was the most direct solution, but it was not good for long-term scaling. Every pricing update would require code changes, testing, and deployment.

The second option was to store all pricing settings in the main application database. This was more flexible, but it made the application more coupled to pricing configuration. It also increased complexity around updates, validation, and support.

The third option was to use an external configuration file generated by a separate service. This was the solution we selected.

In the final approach, pricing settings were managed in an admin panel. A separate service generated a settings file with the needed pricing configuration for merchants. This file was then deployed through CloudFront CDN. The Node.js application loaded this file and used it to apply the correct pricing rules.

I selected this approach because it had a few important advantages. It separated pricing configuration from the core application logic. It allowed faster pricing updates without changing the main codebase. It also used CDN delivery, which made access to configuration fast and reliable.

When settings were changed in the admin panel, the CI/CD pipeline deployed the updated configuration file. After deployment, the cache was invalidated so the new version became available quickly. This allowed business changes to go live in a controlled and automated way.

For testing, I focused on checking that the correct pricing was loaded for each merchant, that configuration changes were applied correctly after deployment, and that invalid or outdated settings would not break the application. I also tried to keep the configuration structure simple and predictable, so the code was easier to maintain.

For performance, CDN delivery helped reduce latency and removed unnecessary load from the main application. For maintainability, separating configuration from business logic made the system easier to extend and support in the future.
